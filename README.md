# mlx-lm API для macOS

Закрытый репозиторий для сервиса, который запускает `mlx-lm` на этом Mac и предоставляет единый HTTP API для работы с разными MLX-моделями.

Базовая модель проекта на текущем этапе: `mlx-community/Qwen3.5-27B-Claude-4.6-Opus-Distilled-MLX-4bit`. Рабочая среда проекта стандартизирована на локальный `virtualenv` в каталоге `.venv`.

Сейчас репозиторий инициализирован как проектная база: здесь зафиксированы цель, архитектурные ожидания, правила разработки и дальнейший план реализации. Код API будет добавляться поверх этой основы.

## Цель проекта

Проект нужен для локального и полуавтоматического запуска LLM-моделей, совместимых с MLX, на Apple Silicon Mac с единым интерфейсом управления:

- поднимать и останавливать модели по API;
- держать реестр доступных моделей и параметров запуска;
- отправлять запросы на генерацию текста и чат;
- ограничивать параллелизм и контролировать загрузку памяти;
- давать предсказуемую точку интеграции для локальных инструментов и внешних сервисов.

## Базовая модель

Первая целевая модель, на которую нужно ориентировать начальную реализацию API и рантайма:

- Hugging Face repo: `mlx-community/Qwen3.5-27B-Claude-4.6-Opus-Distilled-MLX-4bit`
- тип: 4-bit MLX-квантизация reasoning-модели класса Qwen 3.5 27B
- сценарий: локальный inference на Apple Silicon для сложных задач, где важнее качество рассуждения, чем минимальная латентность
- статус в проекте: дефолтная модель для первого рабочего контура

Практические ограничения, которые нужно считать базовыми для этого репозитория:

- минимально разумная машина: Apple Silicon Mac с `24 GB` Unified Memory;
- предпочтительно: `32 GB+`, если нужен запас под длинный контекст и параллельную обвязку API;
- рекомендуемая локальная версия Python: `3.12`;
- модель крупная, поэтому переключение, прогрев и освобождение памяти нужно проектировать как дорогие операции;
- при первом запуске веса будут скачаны из Hugging Face, поэтому проект не должен пытаться хранить их в git.

Если позже появятся дополнительные модели, именно эта остаётся опорной для smoke-check сценариев, пока явно не принято другое решение.

## Базовый сценарий

Сервис будет жить на этом Mac и выступать как orchestration-слой над `mlx-lm`:

1. клиент запрашивает список доступных моделей;
2. API загружает или активирует нужную модель;
3. клиент отправляет prompt или chat-сообщения;
4. сервис возвращает результат и метаданные исполнения;
5. при необходимости модель выгружается, чтобы освободить RAM/Unified Memory.

## Почему отдельный API-слой

`mlx-lm` хорошо решает задачу инференса, но для постоянной локальной эксплуатации обычно не хватает:

- стандартизованного HTTP API;
- учёта активной модели и очереди запросов;
- предсказуемой конфигурации под конкретный Mac;
- контроля таймаутов, лимитов и логирования;
- точки расширения под OpenAI-совместимые ручки.

Этот репозиторий закрывает именно эти организационные и интеграционные задачи.

## Планируемый стек

Предварительный стек, от которого имеет смысл отталкиваться:

- Python 3.12+
- `virtualenv` c локальной директорией `.venv`
- FastAPI или аналогичный ASGI-фреймворк
- Uvicorn для локального запуска
- `mlx-lm` как движок инференса
- Pydantic Settings для конфигурации
- `asyncio` + контролируемый пул задач для сериализации доступа к модели
- структурированные логи в stdout и/или файл

Если в ходе реализации выяснится, что для устойчивости лучше выделять запуск модели в отдельный worker-процесс, это допустимое изменение. В таком случае архитектурное решение нужно будет зафиксировать в README.

Отдельное проектное решение: по умолчанию использовать обычный `virtualenv`, а не Poetry, Conda или другой менеджер окружений. Если позже понадобится другой инструмент, это должно быть отдельным осознанным изменением.

## Предлагаемая структура репозитория

Текущая структура минимальная, но проект логично развивать в таком виде:

```text
.
├── AGENTS.md
├── README.md
├── .gitignore
├── pyproject.toml
├── app/
│   ├── api/
│   ├── core/
│   ├── models/
│   ├── runners/
│   └── schemas/
├── scripts/
│   ├── bootstrap-venv.sh
│   └── chat-default-model.sh
├── tests/
└── var/
```

Назначение директорий:

- `app/api/` - HTTP-маршруты и слой приложения;
- `app/core/` - конфигурация, логирование, общие зависимости;
- `app/models/` - реестр описаний поддерживаемых MLX-моделей;
- `app/runners/` - адаптеры запуска `mlx-lm`, worker-логика и диспетчеризация;
- `app/schemas/` - Pydantic-схемы запросов и ответов;
- `tests/` - модульные и интеграционные тесты;
- `scripts/` - утилиты для локального обслуживания;
- `var/` - runtime-артефакты, которые не должны попадать в git.

## Локальная среда

Стандартный способ работы с проектом:

```bash
./scripts/bootstrap-venv.sh
source .venv/bin/activate
```

Что делает bootstrap:

- создаёт `.venv`, если его ещё нет;
- обновляет `pip`, `setuptools` и `wheel`;
- ставит проект и dev-зависимости из `pyproject.toml`.

После подготовки окружения базовую модель можно прогнать напрямую через `mlx-lm`:

```bash
./scripts/chat-default-model.sh
```

При необходимости можно переопределить модель через переменную окружения:

```bash
MODEL_REPO=mlx-community/Qwen3.5-27B-Claude-4.6-Opus-Distilled-MLX-4bit ./scripts/chat-default-model.sh
```

Если нужно вручную создать окружение без скрипта, эквивалентный минимальный путь такой:

```bash
python3.12 -m venv .venv
./.venv/bin/python -m pip install --upgrade pip setuptools wheel
./.venv/bin/pip install -e ".[dev]"
```

## Черновой API-контур

Ниже не контракт, а стартовый набор ручек, который покрывает основную эксплуатацию:

- `GET /health` - быстрая проверка доступности сервиса;
- `GET /v1/models` - список известных моделей и их статусов;
- `POST /v1/models/load` - активировать модель с параметрами запуска;
- `POST /v1/models/unload` - выгрузить активную модель;
- `GET /v1/runtime` - текущее состояние рантайма, памяти и очереди;
- `POST /v1/generate` - обычная текстовая генерация;
- `POST /v1/chat/completions` - OpenAI-подобный чат-интерфейс;
- `POST /v1/completions` - совместимый completion-интерфейс, если он понадобится интеграциям.

## Управление моделями

Ожидаемый подход:

- модели описываются декларативно, а не размазываются по коду;
- у каждой модели есть стабильный идентификатор, source/path и presets;
- активной считается одна загруженная модель, если явно не реализован multi-worker режим;
- переключение модели должно быть явной операцией с понятной стоимостью;
- сервис обязан отдавать причину отказа, если модель не помещается в память или уже идёт конфликтующий запуск.

## Конфигурация

Предпочтительный формат конфигурации:

- `.env` для локальной среды;
- переменные окружения для production-like запуска;
- отдельный конфиг-объект приложения с жёсткой валидацией.

Пример ожидаемых переменных:

```env
HOST=127.0.0.1
PORT=8000
LOG_LEVEL=INFO
MODEL_REGISTRY_PATH=./config/models.yaml
DEFAULT_MODEL_HF_REPO=mlx-community/Qwen3.5-27B-Claude-4.6-Opus-Distilled-MLX-4bit
MAX_CONCURRENT_REQUESTS=1
REQUEST_TIMEOUT_SECONDS=300
```

## Требования к реализации

- ориентироваться на Apple Silicon и реальные ограничения Unified Memory;
- не хардкодить личные абсолютные пути в коде;
- считать `virtualenv` в `.venv` стандартным способом локального запуска;
- отделять HTTP-слой от логики запуска моделей;
- не смешивать модельный реестр, настройки рантайма и схемы API;
- предусмотреть безопасную деградацию при нехватке памяти, таймауте или падении процесса;
- проектировать API так, чтобы позже можно было добавить авторизацию без полной переделки.

## Безопасность и эксплуатация

Так как репозиторий закрытый, он подходит для внутренних сценариев, но даже в таком режиме:

- нельзя коммитить токены, ключи и `.env`;
- нельзя хранить в git скачанные веса моделей;
- нужно логировать факты запуска/остановки модели и ошибки исполнения;
- при появлении внешнего доступа нужен хотя бы токен/API key и ограничение bind-адреса.

Дополнительно для текущей базовой модели:

- нужно учитывать, что она тяжёлая даже в 4-bit варианте;
- для первого рабочего релиза безопаснее считать загрузку одной активной модели стандартным режимом;
- любые оптимизации вокруг выгрузки/перезагрузки нужно проверять на реальном поведении `mlx-lm`, а не на предположениях.
