# AGENTS.md

Этот файл задаёт правила для агентов и разработчиков, работающих в репозитории `mlx-lm`.

## Контекст проекта

Репозиторий предназначен для локального сервиса на этом Mac, который управляет запуском MLX-моделей через `mlx-lm` и предоставляет HTTP API для внешних клиентов.

Ключевой контекст:

- целевая машина - Apple Silicon Mac;
- базовая модель по умолчанию - `mlx-community/Qwen3.5-27B-Claude-4.6-Opus-Distilled-MLX-4bit`;
- проект должен учитывать реальные ограничения CPU/GPU/Unified Memory;
- репозиторий закрытый, но секреты всё равно запрещено хранить в git;
- текущая стадия - формирование базы проекта, затем реализация API и рантайма.

## Главная задача агента

При любом изменении агент должен усиливать проект в сторону надёжного локального inference API, а не просто добавлять код ради кода.

Приоритеты:

- предсказуемый запуск и остановка моделей;
- понятный и стабильный API-контракт;
- управляемое потребление ресурсов;
- прозрачная конфигурация;
- простота сопровождения на одном конкретном Mac.

## Что делать в первую очередь

Перед изменениями:

- прочитать `README.md` и не противоречить зафиксированным решениям;
- проверить текущее состояние git и не затирать чужие изменения;
- держать изменения атомарными и привязанными к конкретной цели;
- если архитектура меняется, обновить документацию в той же задаче.

## Ожидаемая архитектура

Если код ещё не написан или структура только формируется, ориентироваться на разделение:

- HTTP/API слой;
- конфигурация и зависимости приложения;
- диспетчер/runner для `mlx-lm`;
- реестр моделей;
- схемы запросов и ответов;
- тесты отдельно от runtime-кода.

Не смешивать в одном модуле:

- роуты и низкоуровневый запуск процесса;
- чтение env и бизнес-логику;
- описание моделей и формат HTTP-ответа.

## Инженерные ограничения

- не хардкодить абсолютные пользовательские пути;
- не коммитить веса моделей, кэши, логи, `.env` и временные файлы;
- не предполагать, что можно безопасно держать несколько тяжёлых моделей одновременно;
- не делать API молча блокирующим без явного статуса/таймаута;
- не завязывать код на интерактивные ручные шаги, если это можно автоматизировать.

Также:

- стандартная локальная среда проекта - `virtualenv` в `.venv`;
- не переводить проект на Poetry, Conda, uv-managed workflow или другой менеджер без явного запроса;
- если меняется дефолтная модель или способ её запуска, обновить `README.md` в той же задаче.

## Предпочтения по реализации

- Python 3.12+ как основной язык проекта;
- явные типы и валидация входных данных;
- конфигурация через переменные окружения и централизованный settings-модуль;
- ошибки рантайма оформлять так, чтобы по ним можно было понять причину отказа без чтения исходников;
- логи должны помогать разбирать запуск модели, остановку, таймауты и отказ по ресурсам.

Для текущей базовой модели особенно важно:

- помнить, что загрузка модели дорогая по памяти и времени;
- учитывать, что reasoning-ответы могут быть длиннее и тяжелее обычного чата;
- не оптимизировать поведение `<think>`-режима вслепую, пока это не подтверждено тестами и реальным запуском.

## Работа с API

Если агент добавляет или меняет API:

- сохранять единообразные схемы запросов и ответов;
- документировать новые ручки в `README.md`;
- явно отмечать несовместимые изменения;
- по возможности проектировать ручки так, чтобы позже можно было добавить OpenAI-совместимый слой без дублирования логики.

## Работа с моделями

Если агент добавляет поддержку моделей:

- описывать модели декларативно;
- хранить идентификаторы моделей стабильно;
- отделять метаданные модели от параметров конкретного запуска;
- учитывать стоимость переключения модели и освобождения памяти;
- добавлять guard rails против запуска неподдерживаемой или слишком тяжёлой конфигурации.

Пока в проекте не появился полноценный реестр, дефолтным идентификатором считать `mlx-community/Qwen3.5-27B-Claude-4.6-Opus-Distilled-MLX-4bit`.

## Тесты и проверка

Минимальное ожидание для существенных изменений:

- запуск релевантных тестов или, если тестов ещё нет, хотя бы базовой проверки импорта/старта;
- фиксация в ответе, что именно было проверено;
- если проверка не выполнена, это нужно прямо указать.

Когда появится код API, приоритет тестов такой:

1. конфигурация и валидация;
2. состояние рантайма и переходы lifecycle;
3. API-схемы и коды ошибок;
4. интеграции с `mlx-lm` через адаптеры/моки.

## Документация

README должен оставаться актуальным. Агент обязан обновить его, если меняется что-то из этого:

- способ запуска;
- переменные окружения;
- структура каталогов;
- API-контракт;
- модель исполнения или ограничения проекта.

## Git и публикация

- сообщения коммитов писать на русском языке;
- коммиты делать предметными, без сваливания несвязанных изменений;
- перед пушем проверять `git status`;
- если настроен `tea`, считать репозиторий приватным по умолчанию, если не указано иное.

## Нежелательные решения

Избегать:

- гигантских "стартовых" коммитов без понятной структуры;
- скрытых зависимостей от локальной ручной настройки;
- неочевидных фоновых процессов без контроля состояния;
- смешения документации, конфигурации и runtime-кода в одной куче;
- изменений "на будущее" без явной текущей пользы.

## Если информации не хватает

Если задача допускает разумное предположение, агент должен принять его и зафиксировать в ответе. Останавливать работу вопросом стоит только там, где ошибка выбора создаст дорогую или труднообратимую проблему.
